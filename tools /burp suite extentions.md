# extentions:

- logger++
- Autorize
- ActiveScan++
- Collaborator
- Turbo Intruder



افزونه‌های Burp Suite واقعاً سرعت و دقت تست نفوذ رو بالا می‌برن، مخصوصاً برای پیدا کردن باگ‌های P1 (Critical). بذار مهم‌ترین‌هاشون رو برات یکی‌یکی توضیح بدم:

🔹 Logger++

مثل یه Wireshark داخل Burp عمل می‌کنه.

همه‌ی Request/Response ها رو لاگ می‌گیره (حتی وقتی توی Target یا HTTP history حذف بشن).

برای تحلیل جریان درخواست‌ها، Session Hijacking، تست IDOR و بررسی داده‌ها خیلی به‌کار میاد.

معمولاً به صورت پایه‌ای برای تحلیل‌های اولیه استفاده میشه، ولی مستقیم به P1 ربط نداره؛ بیشتر کمکیه.

🔹 Autorize

یکی از پراستفاده‌ترین افزونه‌ها برای باگ‌های P1.

وظیفه: تست Broken Access Control / IDOR / BOLA.

طوری کار می‌کنه که وقتی لاگین هستی و یه درخواست می‌فرستی، همون درخواست رو با توکن/کوکی کاربر دیگه یا بدون لاگین هم می‌فرسته.

اگر جواب یکی بود ➝ یعنی دسترسی درست کنترل نشده (خطرناک، معمولاً P1).

بیشتر توی Bug Bounty برای IDOR / BOLA / Privilege Escalation استفاده میشه.

🔹 ActiveScan++

نسخه‌ی تقویت‌شده‌ی اسکنر داخلی Burp.

Payloadهای اضافه برای تست XSS، SQLi، Command Injection و...

بیشتر برای پیدا کردن Injectionهای High یا Critical استفاده میشه.

البته چون اسکن فعاله، توی برنامه‌های Bug Bounty همیشه توصیه نمی‌شه (بعضی برنامه‌ها فقط Passive Scan می‌خوان).

بیشتر برای SQLi, RCE, Deserialization کمک می‌کنه.

🔹 Collaborator Everywhere

افزونه‌ای برای استفاده از Burp Collaborator.

درخواست‌ها رو دستکاری می‌کنه تا ببینه سرور با یه سرور خارجی تعامل می‌کنه یا نه.

کمک به کشف SSRF, Blind XSS, Blind SQLi, OOB RCE.

خیلی مهمه برای باگ‌های P1 که Blind هستن و نشونه‌ای توی Response نمی‌دن.

مثال: درخواست DNS یا HTTP به دامنه Collaborator = آسیب‌پذیری P1 (مثلاً SSRF).

🔹 Turbo Intruder

📌 وظیفه: فازینگ و بروت‌فورس سریع‌تر از Intruder معمولی

با پایتون و multithreading کار می‌کنه → هزاران درخواست در ثانیه.

ایده‌آل برای Brute-force, Race Condition, Token guessing.

کاربرد عملی:

تست Rate Limit و OTP brute-force.

پیدا کردن Race Condition در پرداخت‌ها یا درخواست‌های critical.

فازینگ سنگین روی APIها.


✅ جمع‌بندی – برای باگ‌های P1 بیشتر این‌ها استفاده میشن:

Autorize → برای Broken Access Control / IDOR / BOLA (خیلی حیاتی)

Collaborator Everywhere → برای SSRF / Blind XSS / Blind SQLi / RCE

ActiveScan++ → برای SQLi / Command Injection / XSS (کمک‌اسکنر قوی)

Logger++ → پشتیبانی و تحلیل (نه مستقیم برای P1، ولی ابزار پایه‌ایه)

# 🔑 افزونه‌های مهم Burp Suite و کاربردهاشون

| افزونه (Extension)       | کاربرد اصلی                                                    | نوع باگ‌های مرتبط            | شدت (احتمال P1)       |
|---------------------------|---------------------------------------------------------------|-------------------------------|------------------------|
| **Logger++**              | لاگ‌برداری کامل از Request/Response، تحلیل جریان داده‌ها       | Session Issues, IDOR کمک‌کننده | ⚪ بیشتر P3/P4 (پشتیبانی) |
| **Autorize**              | تست خودکار **Authorization / Access Control**                 | IDOR, BOLA, Privilege Esc.    | 🔴 خیلی زیاد (P1)       |
| **ActiveScan++**          | اسکن فعال پیشرفته با Payloadهای اضافه                          | SQLi, XSS, RCE, Injections    | 🟠 گاهی P1 (بسته به هدف) |
| **Collaborator Everywhere** | تزریق Payload برای تست تعامل با سرور خارجی (OOB Testing)      | SSRF, Blind XSS, Blind SQLi, RCE | 🔴 خیلی زیاد (P1)       |

---

## 📌 خلاصه
- **Autorize** → بهترین ابزار برای کشف باگ‌های **Broken Access Control (IDOR/BOLA)**.  
- **Collaborator Everywhere** → عالی برای باگ‌های **Blind / OOB (مثل SSRF, Blind XSS)**.  
- **ActiveScan++** → کمک به کشف اینجکشن‌ها و آسیب‌پذیری‌های High/Critical.  
- **Logger++** → ابزار پایه‌ای برای تحلیل و بررسی جریان درخواست‌ها (کمکی، نه مستقیم P1).

🛠 نصب افزونه‌ها در Burp Suite

برو به منوی Extender → BApp Store.

اسم افزونه (مثلاً Autorize یا Logger++) رو جستجو کن.

روی Install بزن → به صورت خودکار نصب میشه.

بعد از نصب، معمولاً تب مخصوص خودش رو توی Burp می‌بینی (مثلاً تب Autorize یا Logger++).

📌 نحوه استفاده و پیدا کردن آسیب‌پذیری‌ها
🔴 1. Autorize (برای Access Control / IDOR / BOLA)

چی کار می‌کنه؟
هر درخواست تو رو دوباره با یک کوکی/توکن دیگه (یا بدون لاگین) می‌فرسته.

چطور استفاده کنی؟

یه اکانت Admin و یه اکانت User معمولی بساز.

کوکی اکانت User رو کپی کن و توی تب Autorize بذار.

حالا Burp وقتی درخواست با اکانت Admin می‌فرستی، همون رو با کوکی User هم می‌فرسته.

اگه پاسخ‌ها یکی بود (یا اطلاعات حساس لو رفت) → Broken Access Control → P1

✅ کاربرد: پیدا کردن IDOR، BOLA، Privilege Escalation.

🔴 2. Collaborator Everywhere (برای Blind باگ‌ها مثل SSRF / Blind XSS)

چی کار می‌کنه؟
هدرها و ورودی‌ها رو دستکاری می‌کنه تا ببینه سرور با سرور خارجی (Collaborator) ارتباط می‌گیره یا نه.

چطور استفاده کنی؟

افزونه رو فعال کن.

درخواست‌ها رو مرور کن (Proxy / Repeater).

اگر توی Burp Collaborator client دیدی یه DNS/HTTP callback برگشت → یعنی آسیب‌پذیری داری.

✅ کاربرد: SSRF، Blind XSS، Blind SQLi، RCE (Out-of-Band).

🟠 3. ActiveScan++ (Injectionها)

چی کار می‌کنه؟
همون اسکنر Burp هست ولی با payloadهای قوی‌تر.

چطور استفاده کنی؟

یه درخواست مشکوک (مثلاً پارامتر id=123) رو بفرست به Active Scan.

ActiveScan++ پارامترها رو با payloadهای خودش تست می‌کنه.

اگر آسیب‌پذیری مثل SQLi یا XSS باشه توی Issues نشون میده.

✅ کاربرد: SQL Injection، Command Injection، XSS، Deserialization.

⚪ 4. Logger++ (پشتیبان و تحلیلگر)

چی کار می‌کنه؟
همه‌ی Request/Responseها رو ذخیره می‌کنه (حتی اگه توی HTTP history پاک بشن).

چطور استفاده کنی؟

برو تب Logger++ → همه‌ی درخواست‌ها لاگ میشن.

می‌تونی فیلتر کنی (مثلاً فقط درخواست‌های POST).

برای پیدا کردن پارامترهای حساس، Session hijacking، تحلیل IDOR خیلی مفیده.

✅ کاربرد: پشتیبانی و تحلیل. (مستقیم P1 نیست، ولی کمک زیادی می‌کنه).

🟢 سناریو ۱: کشف IDOR با Autorize
🎯 هدف

یک سایت فروشگاه اینترنتی که دو نوع کاربر داره:

User عادی (می‌تونه فقط سفارش خودش رو ببینه)

Admin (می‌تونه سفارش همه رو ببینه)

🛠 مراحل

با اکانت Admin لاگین کن.

وقتی میری به /orders/1234 → می‌بینی سفارش کاربر دیگه رو.

Burp → تب Autorize رو باز کن.

کوکی/توکن کاربر User رو بذار داخلش.

حالا وقتی با اکانت Admin روی /orders/1234 کلیک می‌کنی، Autorize همون درخواست رو با کوکی User می‌فرسته.

نتیجه:

اگر User هم تونست جواب مشابه بگیره (سفارش کسی دیگه رو ببینه) → Broken Access Control (IDOR / BOLA) → P1

در Autorize لیست درخواست‌ها رو می‌بینی، رنگ قرمز/سبز نشون میده تفاوتی هست یا نه.

✅ توی باگ‌بانتی: همین میشه یه گزارش P1 خیلی ارزشمند.

🟢 سناریو ۲: کشف SSRF با Collaborator Everywhere
🎯 هدف

یک اپلیکیشن فرم آپلود داره که توش باید URL عکس پروفایل بدی:
POST /upload

```
{ "image_url": "https://example.com/avatar.jpg" }
```
🛠 مراحل

افزونه Collaborator Everywhere رو فعال کن.

این افزونه به صورت خودکار توی هدرها و پارامترها payloadهای دامنه‌ی Burp Collaborator رو تزریق می‌کنه.

درخواست /upload رو با پارامتر image_url ارسال کن.

حالا برو به Burp → Collaborator Client.

اگر درخواست DNS یا HTTP به دامنه‌ی Collaborator برگشت → یعنی سرور رفته URL رو ریزالو کرده.

نتیجه:

یعنی میشه به سرور دستور داد هر آدرسی رو باز کنه (مثلاً http://169.254.169.254/ برای metadata AWS).

این یعنی SSRF → P1.

✅ توی باگ‌بانتی: SSRF خیلی خطرناکه چون ممکنه منجر به RCE یا Data Leak بشه.

افزونه‌های مهم Burp Suite
Logger++
این افزونه قابلیت ضبط و نمایش دقیق لاگ‌های ترافیک HTTP/HTTPS را دارد و به شما امکان می‌دهد تمام درخواست‌ها و پاسخ‌های مرورگر را با جزئیات کامل ببینید. این افزونه برای پیدا کردن آسیب‌پذیری‌ها هنگام بررسی ترافیک بسیار مهم است و در شناسایی رفتارهای مشکوک و غیرمعمول کمک می‌کند.

Autorize
افزونه Autorize برای تست‌های کنترل دسترسی استفاده می‌شود. این افزونه به صورت خودکار درخواست‌ها را تغییر می‌دهد تا مواردی مانند IDOR یا آسیب‌پذیری‌های کنترل دسترسی را بیابد. به این صورت که سعی می‌کند دسترسی به منابع و صفحات محدود شده را با دستکاری درخواست‌ها بررسی کند.

ActiveScan++
نسخه پیشرفته ابزار Active Scanner است که به‌صورت فعال به جستجوی آسیب‌پذیری‌ها در درخواست‌ها می‌پردازد. این افزونه، امکان پویش دقیق‌تر و جامع‌تر آسیب‌پذیری‌های رایج امنیتی مانند SQLi، XSS، و دیگر تزریق‌ها را فراهم می‌کند.

Burp Collaborator
سرویس آنلاین Burp Collaborator نقش مهمی در شناسایی آسیب‌پذیری‌ها دارد که نیاز به تعامل با سرورهای خارجی مانند تست SSRF، Blind XSS، و برخی آسیب‌پذیری‌های out-of-band دارد. این سرویس به‌صورت خودکار درخواست‌های شبکه‌ای را رصد می‌کند که بدون پاسخ مستقیم هستند اما از طریق سرورهای دیگر قابل کشف‌اند.

بیشترین استفاده در کشف آسیب‌پذیری‌های P1
برای مشکلات بحرانی P1 معمولاً ترکیبی از Logger++ برای ضبط و بررسی دقیق ترافیک، ActiveScan++ برای اسکن فعال، و Burp Collaborator برای شناسایی آسیب‌پذیری‌های out-of-band استفاده می‌شود. افزونه Autorize نیز اگر نیاز به تست کنترل دسترسی باشد، بسیار کاربردی است.

نحوه استفاده و پیدا کردن آسیب‌پذیری‌ها با افزونه‌ها
ابتدا مرورگر خود را طوری تنظیم کنید که تمام ترافیک از طریق Burp Suite عبور کند (پروکسی localhost:8080).

افزونه Logger++ را فعال کنید تا تمام ترافیک‌ها و درخواست‌ها با جزئیات جمع‌آوری شود.

در بخش Proxy، درخواست‌هایی که مشکوک به نقاط ورود پارامتر یا بخش‌هایی هستند که ممکن است آسیب‌پذیر باشند، انتخاب و به بخش Intruder یا Repeater ارسال کنید.

از ActiveScan++ برای انجام اسکن‌های فعال روی درخواست‌ها استفاده کنید. این افزونه به صورت خودکار ضعف‌ها را پیدا و گزارش می‌کند.

از نسخه Collaborator برای آزمایش آسیب‌پذیری‌های Blind و out-of-band استفاده کنید؛ مثلاً وقتی یک پارامتر ممکن است باعث ارسال درخواست به بیرون شود، Collaborator ترافیک آن را رصد می‌کند.

اگر هدف کنترل دسترسی است، افزونه Autorize را نصب و روشن کنید تا درخواست‌های مختلف با تغییر پارامترهای دسترسی ارسال و بررسی شوند.

آسیب‌پذیری‌هایی مثل SQL Injection با تزریق payloadهای مخصوص تست می‌شوند، Cross-Site Scripting با تزریق اسکریپت‌های ساده بررسی می‌شود و گزارش کامل آنها توسط Burp Suite همراه با پیشنهاد رفع مشکل ارائه خواهد شد.

کدام افزونه بیشتر برای P1 استفاده می‌شود؟
بر اساس گزارش‌های باگ بانتی 2025 (مانند YesWeHack و HackerOne)، Autorize و Collaborator بیشترین استفاده را برای یافتن آسیب‌پذیری‌های P1 دارند، زیرا:

Autorize: مستقیماً IDOR و BOLA را هدف قرار می‌دهد، که رایج‌ترین باگ‌های P1 هستند (پاداش‌های 5000-20000 دلار).
Collaborator: برای SSRF و Blind XSS، که تأثیر بالایی دارند و به دلیل ماهیت OOB نیاز به ابزار خاص دارند.
Logger++: برای تحلیل درخواست‌ها و شناسایی الگوهای مشکوک (مانند افشای اطلاعات) مکمل است.
ActiveScan++: برای اسکن خودکار و شناسایی CVEهای خاص (مانند Log4Shell) مفید است، اما کمتر برای P1 دستی استفاده می‌شود.

رتبه‌بندی اهمیت (برای P1):

Autorize: برای IDOR و BOLA.
Collaborator: برای SSRF و Blind XSS.
Logger++: برای تحلیل و شناسایی درخواست‌های غیرمنتظره.
ActiveScan++: برای CVEها و اسکن خودکار.


3. نحوه استفاده از افزونه‌ها و یافتن آسیب‌پذیری‌های P1
در زیر، نحوه استفاده از هر افزونه و چگونگی یافتن آسیب‌پذیری‌های P1 (مانند IDOR، SSRF، XSS) توضیح داده شده است:
3.1 Logger++: نحوه استفاده و یافتن P1

نصب:

به Extender > BApp Store در Burp Suite بروید.
Logger++ را نصب کنید.


نحوه استفاده:

به تب Logger++ بروید.
درخواست‌های HTTP از همه ابزارها (Proxy، Scanner، Repeater) را ثبت کنید.
فیلترها را تنظیم کنید:

Filter by Tool: Extensions یا Scanner را انتخاب کنید.
Filter by String: کلمات کلیدی مانند user_id, token, api_key را جستجو کنید.


خروجی را به CSV صادر کنید برای تحلیل آفلاین.


یافتن P1:

مثال: در یک برنامه، Logger++ درخواست‌های API با پارامتر user_id را نشان داد. هکر درخواست /api/user?user_id=123 را دید و با تغییر به user_id=124، اطلاعات کاربر دیگری را دریافت کرد (IDOR، P1).
روش: از فیلتر regex (مثل user_id|session) برای یافتن endpointهای حساس استفاده کنید. سپس در Repeater تست دستی کنید.


مزیت: شناسایی درخواست‌های غیرمنتظره (مثل درخواست‌های بدون احراز هویت) که به IDOR یا افشای اطلاعات منجر می‌شود.

3.2 Autorize: نحوه استفاده و یافتن P1

نصب:

از BApp Store افزونه Autorize را نصب کنید.
نیاز به Burp Suite Professional دارد.


نحوه استفاده:

در تب Autorize، کوکی‌های دو کاربر را تنظیم کنید:

High-privilege user: کوکی کاربر ادمین.
Low-privilege user: کوکی کاربر عادی.


ترافیک را از Proxy یا Scanner به Autorize هدایت کنید.
Autorize درخواست‌ها را با کوکی‌های کم‌صلاحیت یا بدون کوکی بازپخش می‌کند.
نتایج را بررسی کنید: اگر درخواست با کوکی کم‌صلاحیت پاسخ 200 بدهد، احتمال IDOR یا BOLA وجود دارد.


یافتن P1:

مثال: در یک برنامه، Autorize نشان داد که /api/v1/profile?user_id=123 با کوکی کاربر عادی پاسخ 200 می‌دهد و اطلاعات کاربر دیگری را نمایش می‌دهد (IDOR، P1).
روش: درخواست‌های API با پارامترهای user_id, account_id را هدف قرار دهید. در Repeater، درخواست را با کوکی‌های مختلف تست کنید.


مزیت: سریع‌ترین ابزار برای یافتن IDOR و BOLA، که پاداش‌های بالایی دارند.

3.3 ActiveScan++: نحوه استفاده و یافتن P1

نصب:

از BApp Store افزونه ActiveScan++ را نصب کنید.
نیاز به Burp Suite Professional.


نحوه استفاده:

در Scanner > Scan Configuration، ActiveScan++ را فعال کنید.
یک اسکن فعال (Active Scan) روی دامنه هدف اجرا کنید.
نتایج را در Issues بررسی کنید (مانند SSRF، XML Injection).


یافتن P1:

مثال: ActiveScan++ یک SSRF در endpoint /fetch?url=http://internal شناسایی کرد که به سرورهای داخلی دسترسی داشت (P1، پاداش 10000 دلار).
روش: روی endpointهایی با پارامترهای url, endpoint تمرکز کنید. نتایج را با Collaborator تأیید کنید.


مزیت: برای آسیب‌پذیری‌های پیچیده (مانند SSRF، RCE) که نیاز به اسکن خودکار دارند.

3.4 Burp Collaborator: نحوه استفاده و یافتن P1

نصب: داخلی در Burp Suite Professional (نیازی به نصب جداگانه نیست).
نحوه استفاده:

به Collaborator در Burp بروید و یک URL منحصربه‌فرد (مثل xyz.burpcollaborator.net) تولید کنید.
این URL را در پارامترهای مشکوک (مانند url, callback) تزریق کنید.
در Collaborator Client، تعاملات (DNS, HTTP) را بررسی کنید.


یافتن P1:

مثال: تزریق http://xyz.burpcollaborator.net در /api/fetch?url= منجر به درخواست DNS به سرور Collaborator شد (SSRF، P1).
روش: پارامترهای url, redirect را در Repeater با URL Collaborator تست کنید. اگر تعامل ثبت شد، SSRF یا Blind XSS تأیید می‌شود.


مزیت: بهترین ابزار برای باگ‌های OOB مانند SSRF و Blind XSS.


4. نحوه یافتن آسیب‌پذیری‌های P1 با افزونه‌ها
برای یافتن سریع آسیب‌پذیری‌های P1 (مانند IDOR، SSRF، BOLA)، ترکیبی از افزونه‌ها و تست دستی استفاده می‌شود:

IDOR (با Autorize):

در Logger++، درخواست‌های API با user_id یا account_id را پیدا کنید.
در Autorize، درخواست را با کوکی کاربر دیگر تست کنید.
اگر پاسخ 200 و داده حساس نمایش داده شد، در Repeater تأیید کنید.
گزارش: PoC با درخواست و پاسخ (مثل افشای PII).


SSRF (با Collaborator):

در Logger++، endpointهای url, callback را شناسایی کنید.
در Collaborator، URL منحصربه‌فرد تولید کنید و در Repeater تزریق کنید.
تعاملات DNS/HTTP را چک کنید.
گزارش: نشان دهید که سرور داخلی یا ابرداده (مثل AWS metadata) در دسترس است.


Blind XSS (با Collaborator):

در Logger++، فرم‌ها یا پارامترهای comment, message را پیدا کنید.
Payload مانند <script src="http://xyz.burpcollaborator.net"></script> را تزریق کنید.
تعاملات را در Collaborator بررسی کنید.
گزارش: نشان دهید که کد اجرا شده است.


Logic Flaws (با Logger++ و Repeater):

در Logger++، درخواست‌های پرداخت (quantity, price) را پیدا کنید.
در Repeater، مقادیر را به منفی یا صفر تغییر دهید.
اگر تراکنش غیرعادی (مثل افزایش اعتبار) رخ داد، P1 است.
گزارش: PoC با تأثیر مالی.
